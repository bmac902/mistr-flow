# Mistr Flow — Context

A Windows dictation app that transcribes speech, lightly polishes the result, and pastes it into the active application.

## Terms

## Glossary

**Session** — one press-to-release cycle of the hotkey: from hold-to-record start, through recording, transcription, and Polish, to paste (or error/cancel).

**Raw Transcript** — the unmodified output of the speech-to-text engine for a Session. Never shown to the user in v1; exists only as Polish's input.

**Capture** — one hotkey-triggered screenshot event: an image of the active window plus the context it was taken in, treated as evidence to be routed into a live agent conversation, not as pixels to be filed. Named "Capture," not "Screenshot," because the point is the routing, not the image file.

**Eligible Target** — a Herdr pane that may appear in the Capture picker: Herdr identifies it with an `agent` label AND an actionable agent status (idle or working). Presence of optional `agent_session` metadata is NOT the test — recognized agent panes can lack it. Bare/unknown shells and completed/dead panes are never Eligible Targets. Menu labels carry the agent state so a working agent can be knowingly left alone.

**Polish** — the cleanup step that turns a Raw Transcript into the text that gets pasted. Scope is deliberately narrow:
- DOES: fix punctuation and grammar, format spoken lists into actual list structure.
- DOES NOT: remove, reorder, or merge content based on inferred self-correction ("scratch that," restating a thought, etc.) — every word the speaker said survives into the pasted text, even if they misspoke. No semantic judgment calls, no inferred intent beyond punctuation/grammar/lists.
- DOES NOT: rewrite tone, vocabulary, or voice. Casual stays casual. No LinkedIn-ification.
- Rationale: the courier (Polish) must not overthink things. Anything beyond mechanical cleanup risks silently dropping content the user wanted kept — a stricter rule was preferred over a smarter one.

## Decisions

- **Capture v1 cut line (revised twice)**: v1 ships the full loop — hotkey → active-window capture → picker → delivery — as a **direct Mistr Flow → local Herdr integration**: MF asks the local Herdr instance for eligible agent panes, renders them as numbered menu entries, and delivers the capture to the selected pane through the Herdr integration so the chosen agent can inspect the image. One local fallback verb remains: Clipboard. Two earlier framings were rejected: (1) `Add to issue` as the payoff verb — wrong-premise; Blair has never posted screenshots to GitHub issues; 75–90% of real screenshot traffic goes into a live Herdr agent pane. (2) A Control-Room-mediated architecture (capture POSTed to CR, CR enriches and serves the action menu, "zero MF releases per new verb") — rejected for v1 as solving a bigger problem than the actual one, which is copy/paste friction into one concrete local destination. No CR HTTP service, bearer tokens, multipart protocol, or CR deployment in v1. If the delivery spike demonstrates a concrete limitation that direct integration cannot safely solve, a CR adapter may be reconsidered.
- **Capture delivery is an unproven mechanism**: do not assume terminal image paste works. The delivery mechanism (native attachment/paste vs. a persisted local image plus an injected agent-readable reference) is a required **MF/Herdr technical spike** with its own acceptance criterion: user-visible agent access to the screenshot in the selected pane. The spike must prove real agent-visible delivery per eligible agent type, durable target-identity behavior, and whether a reliable MRU/focus signal exists; it also owns capture retention/TTL cleanup of transient working files and measures pane-query latency (which decides the picker's render choreography).
- **Capture delivery execution**: delivery goes through MF's local Herdr integration — Herdr is co-located with MF by definition, so no remote worker or relay exists in v1. MF gains a deliberately thin Herdr adapter (panes, agent labels/status, delivery) but still learns no git/session/issue semantics. MF's paste/keystroke machinery stays dictation-only: delivery is never focus-stealing keystroke injection into a pane.
- **Auto-attach requires the receiving agent to be idle — confirmed 2026-07-15 by controlled live test** (was: an unproven hypothesis): the *same* PNG was delivered twice via `herdr agent send`. Into an **idle** pane it became a real image attachment; into a **working** pane it landed as inert **plain text** and *stayed* plain text even after the human submitted it. So detection runs when the text **arrives**, not on submit — a mid-turn delivery can never recover. This does not break the core guarantee: the agent can always `Read` the exact delivered path.
- **Herdr does nothing with images** (corrected 2026-07-15 — the earlier belief was invented and never checked against the real CLI): the prior note credited "Herdr's own image-path detection" for the upgrade. That is false. Herdr's API schema contains **no** occurrence of image/attach/multimodal/paste/upload/media, and `agent send` is literally `<target> <text>` — Herdr transports text, full stop. The upgrade is done by the **receiving agent CLI** (Claude Code detecting a path in its own input). This matters for aim: auto-attach is a *Claude Code input-handling* property, not a Herdr one, so no Herdr-side flag, command, or version will ever change it. Same failure mode as #27's invented schema — a plausible mechanism nobody ran against the real thing.
- **Open product question — wait-for-idle before delivering?** Herdr exposes `herdr agent wait <target> --status idle --timeout MS`, so MF *could* wait for the chosen pane to go idle before injecting, buying reliable auto-attach at the cost of delivery latency — against a 3s delivery-ack deadline, and an agent mid-long-turn may not go idle for minutes. Not decided, not built. The picker deliberately offers both `idle` and `working` panes (`ACTIONABLE_STATUSES`), so a user can knowingly deliver into a working pane and get plain text plus a `Read`-able path.
- **`focusOnDeliver` config flag** (2026-07-15): after a successful delivery, MF can optionally focus the delivered-to Herdr pane, so the user's cursor lands there ready to type — matching Blair's actual workflow (screenshot → paste into a terminal → immediately add context and send). This is a deliberate, user-chosen exception to "never steal focus," not a reversal of it: off by default (`focusOnDeliver` in `config.json`, `false` unless set `true`), and it only fires after content has already been delivered — the delivery mechanism itself still never needs focus. A focus failure is best-effort and never turns a successful delivery into a failure.
- **Focusing a Herdr pane takes two halves, not one** (solved 2026-07-15, live; see ADR 0002): `herdr agent focus <target>` is necessary but produces **no visible effect on its own** — which is why v1's `focusOnDeliver` looked broken. **Herdr owns no OS window**: both `herdr.exe` processes report `MainWindowHandle=0`. It is a TUI painted by a *host terminal* (Windows Terminal), and that terminal's window is the only real one in the chain. So `agent focus` moves focus inside a window nobody raises; MF must also raise the host window to the OS foreground. Herdr's API cannot do this — none of its 172 methods raises a window, and the window isn't Herdr's to raise. Corollary, from Herdr's own server log: `agent.focus` **already cascades** workspace+tab+pane on one call, so the previously-suspected `herdr workspace focus`/`herdr tab focus` are **not** needed.
- **Identifying Herdr's host window requires a minted title** (2026-07-15): Windows Terminal runs every window in **one process**, so pid→window is one-to-many, and `MainWindowHandle` just returns whoever is first in z-order (it changed between two probes minutes apart — it is not an identifier). Two WT windows with identical "PowerShell" titles are indistinguishable from outside. The only durable discriminator is one MF mints itself: set a nonce via `client.window_title.set`, find the window wearing exactly that title, then `client.window_title.clear`. This is the sole reason MF touches Herdr's socket at all (ADR 0002).
- **Capture delivery acknowledgement**: the delivery operation returns a short synchronous ack meaning "delivered to the pane" — never "agent finished processing." A timeout or ambiguous result means delivery-unknown (not success, not failure) and permits retry; delivery is idempotent (keyed per capture + target) so a retry can never inject the same capture twice.
- **Capture target identity**: menu entries bind to a durable Herdr session/terminal identity, not a compact positional pane id — panes renumber and move; sessions persist. Concretely (corrected 2026-07-15, live testing): that identity is `terminal_id` specifically. `agent_session.value` looks equally plausible as a durable id but Herdr's CLI rejects it outright (`agent_not_found`) — it is not a valid addressing target for any herdr command found so far, only `terminal_id` is.
- **Capture picker menu contract**: strict eligibility (see *Eligible Target* in the glossary) — agents only, never bare shells. At most 8 pane targets (digit slots 2–9; slot 1 is pinned local Clipboard). Each entry carries a human label including agent state, plus an opaque durable target id that MF passes back to the Herdr integration verbatim. Ordering is NOT promised as most-recently-interacted until the delivery spike proves Herdr exposes a reliable MRU/focus signal — `pane list` exposes agent/status/focused but no interaction timestamp. The spike must also prove each eligible agent type can actually consume the delivered image/reference.
- **Verb mutual exclusion**: strict — one modal gesture at a time, no queueing, no preemption. A hotkey refused because the other verb is mid-flight gets a *visible mascot refusal* ("one thing at a time, sir"), never a silent no-op. The delivering state is un-cancellable: Esc does nothing once the action POST is in flight, because delivery is idempotent and timeout already means delivery-unknown — a cancel UI there would lie. Safety rule outranks the joke: active dictation is never interrupted and spoken words are never lost to a capture gesture.
- **Personality is a product property**: Mistr Flow being funny is intentional, not incidental polish. New Capture states (refused, picker, delivering, delivered, failed, unknown) get expressive, concise mascot reactions designed via Claude Design — fun must never hide the actual state or make an error ambiguous.
- **Capture picker surface**: the picker renders inside the existing overlay window — no second UI surface. The window grows upward with the existing card anchored in place, clamps to the display work area, and returns exactly to its resting bounds on dismiss or failure. Non-focus-stealing is preserved; digit/Esc input arrives via temporary global shortcuts (same pattern as the dictation Escape handler). Capture states — picker, delivering, delivered, failed/unknown, cancel — get explicit designs from Claude Design (the same source as the existing storyboard's personality/visual language) and are slotted into the card's design language, not invented ad hoc.
- **Capture picker render — two-phase**: the picker opens on the first frame with pinned slot 1 (Clipboard) already usable; "summoning targets…" is a designed mascot beat while MF queries Herdr for panes; pane entries append as slots 2–9 and their digit shortcuts are registered atomically with the rendered entries — an early `2` can never act on a stale or default target. The pane-query response is bound to its picker/capture instance: after Esc/dismiss or the 2s deadline, a late response is ignored and never resurrects or mutates a closed picker (the TTL sweep reclaims any transient files). Herdr-unavailable renders a concise explicit local-only state (Clipboard + Esc), never a misleading success. Open detail: if the spike measures local pane-query latency comfortably under ~100ms, the two-phase beat may collapse to a single render — the two-phase decision was made under CR-round-trip latency assumptions that no longer hold.
- **No Save verb in v1**: there is no explicit save action and no permanent MistrFlow screenshot archive — Blair virtually never chooses to save screenshots. `1 Clipboard` puts the PNG on the Windows clipboard as an image (not a file path); it is the only local verb. Captures exist solely as transient local/CR working files needed for upload, delivery, retry, and the selected action, then receive ordinary TTL cleanup — nothing is TTL-exempt. When Herdr is unavailable the picker is Clipboard plus Esc/cancel, not a fake archival option.
- **Capture mechanism — PowerShell/Win32 helper**: active-window capture shells out via the established `powershell -NoProfile` house pattern (no new unsigned native binary — the Defender/keylogger incident set that precedent). PrintWindow-first with DPI awareness as explicit acceptance criteria; content protection on the overlay is a fallback flag, never the default. Screen-rect fallback is only acceptable when the target is visibly capturable and must not include MF's own overlay. Spike checks: (1) HWND, title/process metadata, and pixels must bind to the same foreground-window snapshot taken before the picker appears; (2) the PNG is validated non-empty/non-black — a bad grab yields a truthful capture-failed state, never routed bad evidence; (3) tested on both machines against mixed-DPI, GPU-rendered/UWP, minimized, and elevated/protected windows. Temp image lifecycle/cleanup follows the capture TTL rules.
- **Herdr integration mechanism — CLI shell-out, with one documented exception**: MF talks to Herdr by invoking the `herdr` CLI via `execFile` (the house pattern for every OS integration). The **single exception** (ADR 0002, amending ADR 0001) is `client.window_title.set`/`.clear`, which have no CLI equivalent and are the only way to identify Herdr's host terminal window; those two calls go over the documented socket from `src/herdrSocket.ts`. ADR 0001's original rationale — "the socket protocol carries no compatibility promise" — did not survive the evidence: the socket is publicly documented, versioned (`protocol: 16`), ships a JSON Schema (`herdr api schema`), and **the CLI is itself a socket client** (Herdr's log labels CLI traffic `cli:agent:focus method="agent.focus"`), so it is a wrapper, not a stabler contract. Widening the exception needs a new decision. The adapter's version/capability check runs against the CLI at init. Subprocess latency is measured by the delivery spike and sits well inside the 2s pane-query deadline.
- **Herdr's socket has two live quirks any client must handle** (2026-07-15): (1) the **first connect after idle gets a dead pipe instance** — attempt #1 fails with EPIPE, #2 succeeds, so a single-shot client makes a healthy socket look broken (this cost real debugging time); (2) on Windows the socket is a **named pipe whose name is the socket path verbatim** (`\\.\pipe\C:\Users\...\herdr.sock`), while the `.sock` file is a 25-byte `pid:startTimeNanos` rendezvous file, not a socket. MF discovers the path from `herdr status --json` (`server.socket`) rather than reconstructing it from `%APPDATA%`, so `--session`-scoped sockets keep working.
- **Capture/Herdr integration contract**: MF mints one UUID per capture and reuses it across delivery retries; delivery is deduped idempotently, and a reused id whose payload or target doesn't match is *rejected* — never silently delivered wrong. The integration performs an explicit version/capability check against the local Herdr instance (MF and Herdr ship independently on two hosts); a mismatch produces a clear local-only/incompatible state, not a mysterious failure. Failures map to a structured code plus a safe human-facing message ("that pane has left the building"), never raw socket paths or exception text; MF renders the message as text only. Deadlines: 2s pane query, 3s delivery ack; beyond that, delivery-unknown. (The former CR wire contract — HTTP multipart, bearer tokens, `/captures` — is gone with CR itself.)
- **Delivery addressing (corrected 2026-07-15, live testing)**: `herdr pane run` only accepts the compact/positional `pane_id` — it does **not** accept `agent_session.value` or `terminal_id`, contrary to the original design assumption. `herdr agent send <target> <text>` is the command that actually accepts terminal ids as a target, confirmed live. Delivery uses `agent send`, not `pane run`.
- **Delivery does not auto-submit**: `agent send` writes the injected path as literal text only — it never presses Enter. This is deliberate, not a placeholder: "sending a screenshot to a coding agent with no context is not good at all" (Blair, 2026-07-15). The reference lands in the target pane's input box for the human to add context to and send themselves. Auto-send may be revisited later once usage proves it's wanted — not a default to build toward now. This is consistent with the existing "delivered ≠ agent finished" ack semantics: delivered means the reference landed in the pane, not that anyone acted on it.
- **Capture v1 topology — local-first on two hosts**: v1 must provide the complete same-machine loop (capture → picker → local Herdr pane delivery) on BOTH the home and work machines — both already run MF and Herdr; no new service provisioning. Cross-machine routing (capture on one host delivered to a pane on the other) remains a separate, explicit v1 non-goal. Durable target identity and idempotent delivery keep the design compatible with a CR adapter or relay later, if the spike ever forces one.
- **Listening confirmation**: hotkey-down triggers a floating overlay (mascot tips-hat sprite + live waveform) plus a tiny beep, confirming recording has started.
- **Hotkey model**: hold-to-record (push-to-talk). Release = stop recording and send for transcription. Rejected toggle mode for v1.
- **Paste behavior**: auto-paste immediately when Polish finishes — no preview/confirm step.
- **Self-correction handling**: none in v1. Considered trigger-phrase detection ("scratch that") and general LLM judgment; rejected both in favor of doing nothing, to guarantee Polish never drops content the user said.
- **Hotkey**: Ctrl+Win, held (hold-to-record). Hardcoded for v1, matches existing Wispr Flow muscle memory.
- **APIs**: OpenAI end-to-end — Whisper API for transcription, gpt-4o-mini for Polish. Single vendor, uses existing OpenAI API credit. (Note: Claude Pro is a chat subscription, not API credit — not usable here without separate Anthropic API billing.)
- **Cancel**: two paths, both feed the "Cancelled" mascot state (exits stage left). (1) Dead-zone — releasing the hotkey within ~0.3s of pressing is treated as an accidental trigger, auto-cancels, nothing sent. (2) Esc — pressing Esc while still holding the hotkey deliberately aborts a longer recording in progress.
- **Error fallback**: if Polish (cleanup) fails after transcription succeeded, paste the raw transcript instead of nothing — hat still falls off to signal trouble, but spoken words are never silently lost. If transcription itself fails, there's nothing to paste — error toast only.
- **Mascot rendering**: a borderless, always-on-top overlay window using simple looping sprite-sheet animations per state (not single static poses) — e.g. the moustache-wiggle actually wiggles during Recording. No skeletal rigging or animation engine.
- **API key storage**: plain JSON config file at `%APPDATA%\MistrFlow\config.json`. No Credential Manager, no env var requirement.
- **v1 non-goals (explicitly refused)**: dictation history/transcript log (fire-and-forget, nothing persisted beyond config), a settings UI (hand-edit the JSON file), multi-language support (English only).
- **Private vocabulary support**: optional custom vocabulary lives in the local runtime config under `%APPDATA%\MistrFlow\config.json`, not in the repo. An optional `vocabulary` object supports `terms` (names/acronyms), `phrases` (multi-word concepts), and `replacements` (explicit mishearing corrections). Terms and phrases are passed to Whisper as a transcription `prompt`; all three are passed to Polish as correction context. Public docs, tests, issue text, and PR descriptions must use fake sample terms only. The vocabulary is sent to OpenAI during dictation, matching the existing audio/transcript privacy boundary (the boundary is GitHub/local disk, not OpenAI).
- **v1 in-scope**: auto-start on Windows boot is acceptable to include.
- **Tech stack**: TypeScript + Electron + React. Chosen over C#/.NET despite Electron's larger footprint, because the user already knows this stack and low-friction-to-build outweighs architectural leanness for a personal tool. Paste-into-active-app requires a native keystroke-simulation library (e.g. `@nut-tree/nut-js`) alongside Electron's `clipboard` module, since Electron alone can't send keystrokes to another app's window.
- **Night-one prototype scope**: full core loop (Ctrl+Win hold-to-record → Whisper transcription → gpt-4o-mini Polish → clipboard + simulated paste into active app), with placeholder mascot visuals (static shapes/text per state, not real art). Proves the technically risky part end-to-end.
- **Weekend-pass scope**: real animated mascot artwork (sprite-sheet loops per state) replacing the night-one placeholders; auto-start on boot.
- **Paste targeting**: pastes into whatever app is focused at the moment processing/Polish completes — not the window that was active when recording stopped. Known sharp edge (switching windows mid-process can mis-target the paste); accepted as a simplicity trade for v1.
- **Overlay placement**: bottom-center, just above the taskbar, on the monitor with the focused window — matches Wispr Flow's existing placement, which the user is already used to.
- **Overlay placement (revised)**: simplified to always show on the OS-designated primary monitor, not the monitor with focus. Tracking the focused window's monitor was unnecessary complexity for a personal tool — the user just keeps their primary display set to whichever monitor they actually want it on.
- **State timing**: no artificial minimum duration for Processing/Polishing — states last exactly as long as the real API calls take. Not pre-optimizing for a hypothetical "too fast to see" case.
- **Transcription mode**: batch only — record the full clip locally while the hotkey is held, send one complete audio file to Whisper's standard endpoint on release. No streaming/Realtime API.
- **Idle/at-rest presence**: a small black bar floats persistently above the taskbar at all times the app is running (matches Wispr Flow's pattern) — not tray-icon-only. At idle, only the mascot's hat and eyes peek above the bar. Pressing the hotkey grows the same bar into the larger active overlay with a live waveform; the mascot "gets up" and plays its fuller per-state animations (tip-hat, moustache-wiggle, cane-twirl, bow, etc.) during the active session.
- **Quit/controls**: right-click the persistent bar for a context menu (Quit, open config file). No separate system tray icon — the bar is the only UI surface.
- **Hotkey model (revised)**: switched from hold-to-record to toggle (press to start, press again to stop). Reason: true hold/release detection requires a system-wide low-level keyboard hook; the library providing that (`node-global-key-listener`) bundles a native binary that Windows Defender flagged as `HackTool:MacOS/KeyLogger!rfn` (High severity). Replaced with Electron's built-in `globalShortcut`, which only fires on key-down with no release event, making toggle the only option without a flagged dependency. Escape-to-cancel still works, registered only while a session is active. The dead-zone debounce (accidental brief hold) no longer applies and was removed from `main.ts`.
- **Hotkey (revised)**: `Ctrl+Alt+D`, not `Ctrl+Win`. Electron's `globalShortcut` accelerator strings require at least one non-modifier key, ruling out plain `Ctrl+Win` or `Ctrl+Alt` alone. `Ctrl+Win+Space` and `Ctrl+Alt+Space` were both already bound (`Win+Space` is OS-reserved for input-language switching; `Ctrl+Alt+Space` collided with the user's PowerToys setup). `Shift+Z` worked but fires on every capital "Z" typed anywhere, since `globalShortcut` intercepts system-wide regardless of focus — confirmed full loop (toggle → record → Whisper → Polish punctuation/lists → paste) worked end-to-end with it, then swapped to `Ctrl+Alt+D` ("dictate") to avoid the typing collision.
