# Mistr Flow — Context

A Windows dictation app that transcribes speech, lightly polishes the result, and pastes it into the active application.

## Terms

## Glossary

**Session** — one press-to-release cycle of the hotkey: from hold-to-record start, through recording, transcription, and Polish, to paste (or error/cancel).

**Raw Transcript** — the unmodified output of the speech-to-text engine for a Session. Never shown to the user in v1; exists only as Polish's input.

**Capture** — one hotkey-triggered screenshot event: an image of the active window plus the context it was taken in, treated as evidence to be routed into a live agent conversation, not as pixels to be filed. Named "Capture," not "Screenshot," because the point is the routing, not the image file.

**Relay** — one hotkey-triggered read of the Windows clipboard, routed like a Capture: copied code, a stack trace, a URL, terminal output, or an image. The clipboard is Mistr Flow's **third sense organ** (after voice and screen) — same pipeline shape, different input. Read on demand only: MF never watches, polls, or logs the clipboard, and keeps no history. (Named "Relay," not "Clip": *clipboard* already means the pinned slot-1 *destination* in the Capture picker, and a verb that reads it needed a name that couldn't be confused with one that writes it.)

**Herald** — one press of `Ctrl+Alt+H` ("H for Herdr"): a dictation Session whose Polished text is *routed to an agent pane* instead of pasted locally. Voice's front half (record → Polish) joined to Relay's back half (picker → deliver) — the Polished transcript rides the picker's read-only text preview, and picking a target *is* the confirm-and-send. Slot 1 stays "paste here" (the `Ctrl+Alt+D` outcome) as a salvage; panes on 2–9. Distinct from plain **Dictation** (voice → local paste on `Ctrl+Alt+D`, no picker), which Herald leaves untouched. Named for the butler heralding your words to the agent — and it shares three letters with Herdr. See ADR 0003.

**Last Target** — the single most-recently delivered-to *Eligible Target*, shared across **all** send verbs: a successful pane delivery by Capture, Relay, or Herald updates the same one memory. Deliberately not per-verb — "the agent I'm working with right now" is a property of the user's attention, not of the verb that happens to be carrying the payload (75–90% of *all* sends go to that pane, the same evidence behind all three verbs). Slot-1 outcomes (Clipboard, paste-here) never update it: they aren't panes. No time-based expiry (the visible mark, not a clock, is the staleness defense), no restart persistence (in-memory only — fire-and-forget), validated at use: reconciled against the fresh pane query, and a confirm to a since-dead pane fails truthfully through the ordinary delivery failure. See #46.

**Eligible Target** — a Herdr pane that may appear in the Capture picker: Herdr identifies it with an `agent` label AND an actionable agent status (idle, working, or done). Presence of optional `agent_session` metadata is NOT the test — recognized agent panes can lack it. Bare/unknown shells and **dead** panes (vanished from the pane list) are never Eligible Targets. **Dead and done are different concepts** (corrected 2026-07-16 — the earlier "completed/dead panes" wording conflated them): a *done* pane is a live session waiting at its prompt, spike-verified receivable. Done panes are eligible (#76, landed 2026-07-16) — delivery into a done pane is engagement, so Herdr clears its Done signal by itself, the same way typing does; no special case exists for this anywhere. Menu labels carry the agent state so a working or done agent can be knowingly chosen — or left alone.

**Watched Set** — the opposite end of the same `pane list`: every pane carrying an `agent` label, with its status preserved across the *whole* spectrum (idle / working / blocked / done / unknown) rather than filtered to the actionable subset, and uncapped (the whole fleet is in view, not just the 8 picker slots). It shares the picker's two exclusions — bare shells and panes with no durable `terminal_id` are out — but nothing else. This is the seam fleet awareness (#44) reads from; the picker's *Eligible Target* rule is untouched by it.

**Project Anchor** — a per-machine config mapping (`projectAnchors` in config.json) from a cwd path-prefix to a friendly project name and a glyph id: the picker row's WHERE channel. Longest prefix wins (so `…\hermes` and `…\hermes\scratch` coexist), matching is case-insensitive and path-boundary-aware. The glyph *library* (hand-drawn SVGs — never stock emoji, never product logos) ships in the renderer so every machine pulls the capability; each machine's config names only its own projects — MF learns no project semantics. The full row grammar (2026-07-17): **keycap color = WHO** (agent species: claude orange, copilot blue, hermes gold, codex slate; unknown keeps brass), **glyph = WHERE** (project), **text = WHAT** (name · status, raw label preserved as hover title). Three orthogonal channels, none competing.

**Done** — the Watched Set status meaning an agent finished its turn and nobody has engaged with it since: work ready for harvest, never a bottleneck. Herdr owns the entire lifecycle — `done` begins when the agent's turn ends and clears on *engagement* (typing in the pane), **not on focus alone** (spike-verified 2026-07-16: a focused pane sat `done` for minutes until input arrived; `done` and `idle` are distinct states in live practice). Mistr Flow renders the signal exactly as long as Herdr reports it and holds no seen/visited state of its own. The contrast term is **Blocked** — "I need you now," a bottleneck that escalates; Done is "your work is ready, sir" — informational, one soft cue, never nagging.

**Polish** — the cleanup step that turns a Raw Transcript into the text that gets pasted. Scope is deliberately narrow:
- DOES: fix punctuation and grammar, format spoken lists into actual list structure.
- DOES NOT: remove, reorder, or merge content based on inferred self-correction ("scratch that," restating a thought, etc.) — every word the speaker said survives into the pasted text, even if they misspoke. No semantic judgment calls, no inferred intent beyond punctuation/grammar/lists.
- DOES NOT: rewrite tone, vocabulary, or voice. Casual stays casual. No LinkedIn-ification.
- Rationale: the courier (Polish) must not overthink things. Anything beyond mechanical cleanup risks silently dropping content the user wanted kept — a stricter rule was preferred over a smarter one.

## Decisions

- **Capture v1 cut line (revised twice)**: v1 ships the full loop — hotkey → active-window capture → picker → delivery — as a **direct Mistr Flow → local Herdr integration**: MF asks the local Herdr instance for eligible agent panes, renders them as numbered menu entries, and delivers the capture to the selected pane through the Herdr integration so the chosen agent can inspect the image. One local fallback verb remains: Clipboard. Two earlier framings were rejected: (1) `Add to issue` as the payoff verb — wrong-premise; Blair has never posted screenshots to GitHub issues; 75–90% of real screenshot traffic goes into a live Herdr agent pane. (2) A Control-Room-mediated architecture (capture POSTed to CR, CR enriches and serves the action menu, "zero MF releases per new verb") — rejected for v1 as solving a bigger problem than the actual one, which is copy/paste friction into one concrete local destination. No CR HTTP service, bearer tokens, multipart protocol, or CR deployment in v1. If the delivery spike demonstrates a concrete limitation that direct integration cannot safely solve, a CR adapter may be reconsidered.
- **Capture delivery is an unproven mechanism**: do not assume terminal image paste works. The delivery mechanism (native attachment/paste vs. a persisted local image plus an injected agent-readable reference) is a required **MF/Herdr technical spike** with its own acceptance criterion: user-visible agent access to the screenshot in the selected pane. The spike must prove real agent-visible delivery per eligible agent type, durable target-identity behavior, and whether a reliable MRU/focus signal exists; it also owns capture retention/TTL cleanup of transient working files and measures pane-query latency (which decides the picker's render choreography).
- **Capture delivery execution**: delivery goes through MF's local Herdr integration — Herdr is co-located with MF by definition, so no remote worker or relay exists in v1. MF gains a deliberately thin Herdr adapter (panes, agent labels/status, delivery) but still learns no git/session/issue semantics. MF's paste/keystroke machinery stays dictation-only: delivery is never focus-stealing keystroke injection into a pane.
- **Auto-attach requires the receiving agent to be idle — confirmed 2026-07-15 by controlled live test** (was: an unproven hypothesis): the *same* PNG was delivered twice via `herdr agent send`. Into an **idle** pane it became a real image attachment; into a **working** pane it landed as inert **plain text** and *stayed* plain text even after the human submitted it. So detection runs when the text **arrives**, not on submit — a mid-turn delivery can never recover. This does not break the core guarantee: the agent can always `Read` the exact delivered path.
- **Herdr does nothing with images** (corrected 2026-07-15 — the earlier belief was invented and never checked against the real CLI): the prior note credited "Herdr's own image-path detection" for the upgrade. That is false. Herdr's API schema contains **no** occurrence of image/attach/multimodal/paste/upload/media, and `agent send` is literally `<target> <text>` — Herdr transports text, full stop. The upgrade is done by the **receiving agent CLI** (Claude Code detecting a path in its own input). This matters for aim: auto-attach is a *Claude Code input-handling* property, not a Herdr one, so no Herdr-side flag, command, or version will ever change it. Same failure mode as #27's invented schema — a plausible mechanism nobody ran against the real thing.
- **Open product question — wait-for-idle before delivering?** Herdr exposes `herdr agent wait <target> --status idle --timeout MS`, so MF *could* wait for the chosen pane to go idle before injecting, buying reliable auto-attach at the cost of delivery latency — against a 3s delivery-ack deadline, and an agent mid-long-turn may not go idle for minutes. Not decided, not built. The picker deliberately offers `idle`, `working`, and `done` panes (`ACTIONABLE_STATUSES`), so a user can knowingly deliver into a working pane and get plain text plus a `Read`-able path.
- **`focusOnDeliver` config flag** (2026-07-15): after a successful delivery, MF can optionally focus the delivered-to Herdr pane, so the user's cursor lands there ready to type — matching Blair's actual workflow (screenshot → paste into a terminal → immediately add context and send). This is a deliberate, user-chosen exception to "never steal focus," not a reversal of it: off by default (`focusOnDeliver` in `config.json`, `false` unless set `true`), and it only fires after content has already been delivered — the delivery mechanism itself still never needs focus. A focus failure is best-effort and never turns a successful delivery into a failure.
- **Focusing a Herdr pane takes two halves, not one** (solved 2026-07-15, live; see ADR 0002): `herdr agent focus <target>` is necessary but produces **no visible effect on its own** — which is why v1's `focusOnDeliver` looked broken. **Herdr owns no OS window**: both `herdr.exe` processes report `MainWindowHandle=0`. It is a TUI painted by a *host terminal* (Windows Terminal), and that terminal's window is the only real one in the chain. So `agent focus` moves focus inside a window nobody raises; MF must also raise the host window to the OS foreground. Herdr's API cannot do this — none of its 172 methods raises a window, and the window isn't Herdr's to raise. Corollary, from Herdr's own server log: `agent.focus` **already cascades** workspace+tab+pane on one call, so the previously-suspected `herdr workspace focus`/`herdr tab focus` are **not** needed.
- **Identifying Herdr's host window requires a minted title** (2026-07-15): Windows Terminal runs every window in **one process**, so pid→window is one-to-many, and `MainWindowHandle` just returns whoever is first in z-order (it changed between two probes minutes apart — it is not an identifier). Two WT windows with identical "PowerShell" titles are indistinguishable from outside. The only durable discriminator is one MF mints itself: set a nonce via `client.window_title.set`, find the window wearing exactly that title, then `client.window_title.clear`. This is the sole reason MF touches Herdr's socket at all (ADR 0002).
- **Capture delivery acknowledgement**: the delivery operation returns a short synchronous ack meaning "delivered to the pane" — never "agent finished processing." A timeout or ambiguous result means delivery-unknown (not success, not failure) and permits retry; delivery is idempotent (keyed per capture + target) so a retry can never inject the same capture twice.
- **Capture target identity**: menu entries bind to a durable Herdr session/terminal identity, not a compact positional pane id — panes renumber and move; sessions persist. Concretely (corrected 2026-07-15, live testing): that identity is `terminal_id` specifically. `agent_session.value` looks equally plausible as a durable id but Herdr's CLI rejects it outright (`agent_not_found`) — it is not a valid addressing target for any herdr command found so far, only `terminal_id` is.
- **Capture picker menu contract**: strict eligibility (see *Eligible Target* in the glossary) — agents only, never bare shells. At most 8 pane targets (digit slots 2–9; slot 1 is pinned local Clipboard). Each entry carries a human label including agent state, plus an opaque durable target id that MF passes back to the Herdr integration verbatim. Ordering is NOT promised as most-recently-interacted until the delivery spike proves Herdr exposes a reliable MRU/focus signal — `pane list` exposes agent/status/focused but no interaction timestamp. The spike must also prove each eligible agent type can actually consume the delivered image/reference.
- **Verb mutual exclusion**: strict — one modal gesture at a time, no queueing. A hotkey refused because the other verb is mid-flight gets a *visible mascot refusal* ("one thing at a time, sir"), never a silent no-op. **Amended 2026-07-17 — an open picker yields**: a *different* verb's hotkey while a picker is open cancels the picker and starts the intended verb (the "switch" arbiter decision) — one press, no Esc, no trap. The refusal was trapping mis-keyed verbs behind a manual Esc that, mis-aimed, landed in the focused pane and killed live coding sessions. A picker is a menu, not work. Same-verb press while its own picker is open remains the again-confirm (ADR 0004). The delivering state is un-cancellable: Esc does nothing once the action POST is in flight, because delivery is idempotent and timeout already means delivery-unknown — a cancel UI there would lie. **Amended again 2026-07-17 — the double-press escape hatch (layer 2, `refuseOrForceSwitch`)**: a *busy* state (recording, pre-picker grab, transient beats — states with no picker to yield) wags on the *first* foreign press, but the **same intended key pressed again within 3s forces the switch** — a mistaken recording is cancelled through the exact path Esc takes, un-cancellable phases (delivering, mid-polish) run out on their own and the switch lands when the lock frees, spoken words in flight pasted first. This deliberately narrows the old "active dictation is never interrupted / spoken words are never lost to a capture gesture" safety rule, and does so on purpose: a *single* stray press still never loses speech (it only wags), but a **deliberate repeat within 3s is treated as intent, not an accident** — the whole point was to stop trapping the user behind a manual Esc that, mis-aimed, killed live coding sessions. Operator-verified 2026-07-17 (a "smarter" token-supersession guard dogfooded worse and was reverted; a known quirk remains where rapid H-hammering can toggle/restart the fresh recording — livable, revisit with fresh eyes).
- **Personality is a product property**: Mistr Flow being funny is intentional, not incidental polish. New Capture states (refused, picker, delivering, delivered, failed, unknown) get expressive, concise mascot reactions designed via Claude Design — fun must never hide the actual state or make an error ambiguous.
- **Clipboard is a source, not just a destination** (2026-07-15, issues #37–#40): the clipboard becomes a third sense organ — copy anything, hotkey, digit, done — killing the `Ctrl+C → Alt+Tab → find pane → Ctrl+V` dance. It is the verb where the plumbing is *native* rather than coerced: `herdr agent send <target> <text>` transports text, full stop, so none of the image auto-attach caveats (idle-pane requirement, plain-text fallback) apply. Every delivery path reduces to the same primitive — inject a string, optionally after verifying a file exists — which is the seam the send session generalizes over.
- **Relay digit slots stay aligned with Capture's** (2026-07-15; revised 2026-07-16): panes occupy 2–9 across both verbs — the stable-slot muscle-memory principle outranks slot efficiency. Originally slot 1 was *skipped* (Capture's slot 1 is Clipboard-as-destination, judged meaningless when the clipboard is the source). **Revised**: with `copySelectionFirst`, `Ctrl+Alt+C` *is* the copy — so slot 1 returns as **"1 Clipboard" = keep the copy, stop here**: an affirmative local ending (success beat, never the cancelled beat Esc gives), for every source kind including images. The judged-meaningless reading missed that the *copy itself* can be the outcome — select → hotkey → `1` replaces plain Ctrl+C, and the same gesture reaches the panes on 2–9. See "Slot 1 is the local outcome."
- **Long relayed text spills to a file rather than truncating; threshold ~10k chars** (2026-07-15): short text injects inline; over the threshold, MF writes a temp file and injects its path for the agent to `Read`. Rejected truncation — the tail of a stack trace is usually the part that matters, and silently dropping it is the same class of failure as Polish dropping words. Rejected refusing — that hands the worst payloads back to the manual dance. The threshold is set high (~10k chars ≈ 200 lines) *because inline is strictly better UX*: text in the pane's input box can be seen, edited, and wrapped in context before you send it, whereas a path is an opaque black box. So spilling is the fallback for whole files and huge logs, not the normal case. Spill files are transient with the same TTL sweep as captures; no archive, no clipboard history.
- **Relay handles images, which back-doors precise region select** (2026-07-15): `clipboard.readImage()` routes through the identical PNG-path machinery Capture uses. The payoff beyond completeness: **Win+Shift+S → hotkey → digit** delivers a pixel-precise region snip into a pane, using Windows' own snipping tool — which already handles this machine's mixed-DPI, mixed-orientation desktop. That is the precision path the in-picker crop deliberately isn't, obtained for free rather than by building a full-screen selector. Crop-on-a-relayed-image should fall out of the shared artifact type for free; if it turns out to need real plumbing, it gets its own issue rather than bloating the verb.
- **Relay has no local fallback when Herdr is down** (2026-07-15; **reversed 2026-07-16** by slot 1's return): with slot 1 restored, a down Herdr or empty fleet degrades to "Clipboard + Esc" exactly like Capture — the truthful message becomes "your copy is safe on the clipboard," never the now-false "nowhere to send it." The empty-clipboard state (nothing-to-send, no picker) is unchanged. Still rejected: an auto-fading flash — the never-fake-a-success rule stands.
- **Region selection is a crop of the grab, not a snipping pass over the live desktop** (2026-07-15): the driver is needing part of a window — e.g. something sits below an API key you don't want to send. Rejected the classic Win+Shift+S model (dim the live screen, drag, then capture) for one decisive reason: Blair's desktop is a 4K landscape (3840×2160 at 0,0) beside a **portrait** 1440×2560 — a 5280×2560 virtual desktop with mixed orientation and mixed DPI. A transparent click-capturing window spanning that is a coordinate/DPI bug farm. Cropping the grab needs none of it: the PNG is already frozen and full-resolution. The trade is selection precision, bounded by the preview's size — accepted because the crop applies to the **original pixels** (output is full-res; only the gesture is coarse), and because the result is rendered straight back so a mis-drag is seen, not assumed. `CAPTURE_PREVIEW_BOX` is the one constant to grow if it proves too coarse in use.
- **Esc undoes a crop before it dismisses the picker** (2026-07-15): the feared failure is a crop that silently clips the thing you wanted (the last line of a paragraph). So Esc is overloaded — with a crop applied it restores the full capture; only then does it cancel. A mis-drag costs one keypress, never the capture. Cropping is chainable (each drag trims what's on screen), and the original is always one Esc away.
- **Capture picker surface**: the picker renders inside the existing overlay window — no second UI surface. The window grows upward with the existing card anchored in place, clamps to the display work area, and returns exactly to its resting bounds on dismiss or failure. Non-focus-stealing is preserved; digit/Esc input arrives via temporary global shortcuts (same pattern as the dictation Escape handler). Capture states — picker, delivering, delivered, failed/unknown, cancel — get explicit designs from Claude Design (the same source as the existing storyboard's personality/visual language) and are slotted into the card's design language, not invented ad hoc.
- **Capture picker render — two-phase**: the picker opens on the first frame with pinned slot 1 (Clipboard) already usable; "summoning targets…" is a designed mascot beat while MF queries Herdr for panes; pane entries append as slots 2–9 and their digit shortcuts are registered atomically with the rendered entries — an early `2` can never act on a stale or default target. The pane-query response is bound to its picker/capture instance: after Esc/dismiss or the 2s deadline, a late response is ignored and never resurrects or mutates a closed picker (the TTL sweep reclaims any transient files). Herdr-unavailable renders a concise explicit local-only state (Clipboard + Esc), never a misleading success. Open detail: if the spike measures local pane-query latency comfortably under ~100ms, the two-phase beat may collapse to a single render — the two-phase decision was made under CR-round-trip latency assumptions that no longer hold.
- **No Save verb in v1**: there is no explicit save action and no permanent MistrFlow screenshot archive — Blair virtually never chooses to save screenshots. `1 Clipboard` puts the PNG on the Windows clipboard as an image (not a file path); it is the only local verb. Captures exist solely as transient local/CR working files needed for upload, delivery, retry, and the selected action, then receive ordinary TTL cleanup — nothing is TTL-exempt. When Herdr is unavailable the picker is Clipboard plus Esc/cancel, not a fake archival option.
- **Capture mechanism — PowerShell/Win32 helper**: active-window capture shells out via the established `powershell -NoProfile` house pattern (no new unsigned native binary — the Defender/keylogger incident set that precedent). PrintWindow-first with DPI awareness as explicit acceptance criteria; content protection on the overlay is a fallback flag, never the default. Screen-rect fallback is only acceptable when the target is visibly capturable and must not include MF's own overlay. Spike checks: (1) HWND, title/process metadata, and pixels must bind to the same foreground-window snapshot taken before the picker appears; (2) the PNG is validated non-empty/non-black — a bad grab yields a truthful capture-failed state, never routed bad evidence; (3) tested on both machines against mixed-DPI, GPU-rendered/UWP, minimized, and elevated/protected windows. Temp image lifecycle/cleanup follows the capture TTL rules.
- **Herdr integration mechanism — CLI shell-out, with one documented exception**: MF talks to Herdr by invoking the `herdr` CLI via `execFile` (the house pattern for every OS integration). The **single exception** (ADR 0002, amending ADR 0001) is `client.window_title.set`/`.clear`, which have no CLI equivalent and are the only way to identify Herdr's host terminal window; those two calls go over the documented socket from `src/herdrSocket.ts`. ADR 0001's original rationale — "the socket protocol carries no compatibility promise" — did not survive the evidence: the socket is publicly documented, versioned (`protocol: 16`), ships a JSON Schema (`herdr api schema`), and **the CLI is itself a socket client** (Herdr's log labels CLI traffic `cli:agent:focus method="agent.focus"`), so it is a wrapper, not a stabler contract. Widening the exception needs a new decision. The adapter's version/capability check runs against the CLI at init. Subprocess latency is measured by the delivery spike and sits well inside the 2s pane-query deadline.
- **Herdr's socket has two live quirks any client must handle** (2026-07-15): (1) the **first connect after idle gets a dead pipe instance** — attempt #1 fails with EPIPE, #2 succeeds, so a single-shot client makes a healthy socket look broken (this cost real debugging time); (2) on Windows the socket is a **named pipe whose name is the socket path verbatim** (`\\.\pipe\C:\Users\...\herdr.sock`), while the `.sock` file is a 25-byte `pid:startTimeNanos` rendezvous file, not a socket. MF discovers the path from `herdr status --json` (`server.socket`) rather than reconstructing it from `%APPDATA%`, so `--session`-scoped sockets keep working.
- **Capture/Herdr integration contract**: MF mints one UUID per capture and reuses it across delivery retries; delivery is deduped idempotently, and a reused id whose payload or target doesn't match is *rejected* — never silently delivered wrong. The integration performs an explicit version/capability check against the local Herdr instance (MF and Herdr ship independently on two hosts); a mismatch produces a clear local-only/incompatible state, not a mysterious failure. Failures map to a structured code plus a safe human-facing message ("that pane has left the building"), never raw socket paths or exception text; MF renders the message as text only. Deadlines: 2s pane query, 3s delivery ack; beyond that, delivery-unknown. (The former CR wire contract — HTTP multipart, bearer tokens, `/captures` — is gone with CR itself.)
- **Delivery addressing (corrected 2026-07-15, live testing)**: `herdr pane run` only accepts the compact/positional `pane_id` — it does **not** accept `agent_session.value` or `terminal_id`, contrary to the original design assumption. `herdr agent send <target> <text>` is the command that actually accepts terminal ids as a target, confirmed live. Delivery uses `agent send`, not `pane run`.
- **Delivery does not auto-submit**: `agent send` writes the injected path as literal text only — it never presses Enter. This is deliberate, not a placeholder: "sending a screenshot to a coding agent with no context is not good at all" (Blair, 2026-07-15). The reference lands in the target pane's input box for the human to add context to and send themselves. Auto-send may be revisited later once usage proves it's wanted — not a default to build toward now. This is consistent with the existing "delivered ≠ agent finished" ack semantics: delivered means the reference landed in the pane, not that anyone acted on it.
- **Capture v1 topology — local-first on two hosts**: v1 must provide the complete same-machine loop (capture → picker → local Herdr pane delivery) on BOTH the home and work machines — both already run MF and Herdr; no new service provisioning. Cross-machine routing (capture on one host delivered to a pane on the other) remains a separate, explicit v1 non-goal. Durable target identity and idempotent delivery keep the design compatible with a CR adapter or relay later, if the spike ever forces one.
- **Listening confirmation**: hotkey-down triggers a floating overlay (mascot tips-hat sprite + live waveform) plus a tiny beep, confirming recording has started.
- **Herald — voice routed to a pane (2026-07-16, #45)**: dictation gains a second hotkey, `Ctrl+Alt+H`, that Polishes the transcript and routes it to an agent pane through the existing send session, instead of pasting locally; `Ctrl+Alt+D` is untouched. Read-and-pick (read-only transcript preview, choosing a target is the send), slot 1 = paste-here salvage, panes 2–9, Polish always on, no new mascot art (composes existing beats). Full rationale and rejected alternatives in ADR 0003.
- **Same agent again — the verb key confirms to the Last Target (2026-07-16, #46)**: every send verb's picker gains a fast path to the single shared *Last Target* (see glossary): pressing the verb's own hotkey again while its picker is open sends to it — `hotkey → hotkey`, with the payload preview and the marked destination flashed on screen in between (ADR 0003's never-send-silently principle, extended to all verbs). A dedicated "again" row renders from memory on the picker's first frame and visibly reconciles against the pane query; digits 2–9 are untouched. No expiry, no restart persistence, truthful failure on a since-dead pane. Rejected: dedicated hotkey(s), double-tap, modifier. Full rationale in ADR 0004.
- **Slot 1 is the local outcome in every verb (2026-07-16)**: the picker's digit 1 always means "end this locally, no pane" — Capture: copy the grab to the clipboard; Herald: paste the transcript here; Relay: keep the copy that `Ctrl+Alt+C` (with `copySelectionFirst`) already made. Capture and Relay both label it "Clipboard"; Herald labels it "Paste here." Slot-1 outcomes never touch the delivery path, so they never update the *Last Target* — that stays structural. Grammar payoff: one gesture family where `1` is always local, `2–9` are always the same panes, and the verb key repeats to the Last Target.
- **Relay multi-file — every file of a multi-select, as one atomic block (2026-07-16, #66)**: an Explorer multi-select relays *all* its files, read from the clipboard's `CF_HDROP` format (`FileNameW` carries only the first by design — kept as fallback). N paths deliver as ONE newline-joined body (bracketed-paste makes it atomic — one payload, one ack, one retry); **N=1 stays byte-identical** to single-file relay, bare and unbracketed, so path-detection/auto-attach is untouched. All N paths are delivery preconditions — any file vanished since the copy fails the *whole* delivery, naming the missing file — by **basename**: the never-raw-error rule keeps full paths out of human-facing failure text, and full-path identity lives in the preview (the basename-preview rejection below is about previews, not failure messages); never a partial, never silent dropping (the Polish-drops-words rule, applied to files). Preview shows full paths one per line (the directory is the payload's identity), summary `Files · N`. **No enforced cap** — designed for small-N (≤10, Blair's stated ceiling); an absurd list rides the existing spill machinery organically. Precedence unchanged: text > image > file(s). Rejected: per-file injections (partial-failure semantics the ledger has no concept of), basename previews, a hard cap.
- **The open picker is modal — rows are buttons, the butler is the handle (2026-07-16, #47)**: every key-cap row in the picker (digits 2–9, slot 1 where present, the ⟲ again-row) is clickable, and a click *is* a press of that row's key — the same selection event through the same channel, never a parallel implementation. While a picker is open, rows don't participate in window-drag (drag is confined to the butler/header) and the resting bar's click-to-jump is suppressed, restored on close. Rows get renderer-owned button affordances (hover/pressed/cursor). Rejected: threshold-based click-vs-drag arbitration on rows. Full rationale in ADR 0005.
- **Done-awareness — the fleet's second lifecycle signal (2026-07-16)**: fleet awareness gains *Done* alongside *Blocked*, bound **directly to Herdr's `done` agent status** — no working→idle inference, no MF-side visited-tracking; Herdr owns the lifecycle and MF renders it (see glossary *Done*). Spike-confirmed live: `done` fires reliably on turn end, holds without flicker, clears on engagement; PRD #44's "unreliable status" worry was a request-side enum gap only and is retired. The alert grammar is amended from "the persistent-block ding is the only interruption" to **one active cue per lifecycle signal, never repeated, never escalating**: Blocked keeps its posture + escalating ding untouched; Done earns exactly one soft chime (a distinct, gentler sound) at the moment of the transition, once per done-episode, mutable in config, verb-suppressed. **The chime is foreground-gated, precisely**: it fires only when *Herdr's specific host window* is not the OS foreground window at the transition — "your work is ready and you aren't looking at Herdr" — reusing ADR 0002's minted-title host-window identification. A coarse "foreground process is a terminal" probe was rejected with eyes open: multiple terminal windows are routinely open, so the approximation would silently suppress exactly the completion alert the feature exists for. Suppression affects only the audible cue — never the recorded done state or the ambient badge, which reflect every done pane ungated. **The butler's bearing stays a pure function of Blocked** — Done is structurally barred from the posture/tier; its only ambient rendering is a renderer-owned count of *current* done panes on the idle bar (placeholder look now, Claude Design blessing later — the #44 pattern). **No seen-tracking, deliberately**: an "unseen completions" badge was considered and rejected as an unread-message system MF doesn't need — the badge mirrors Herdr's now, nothing more. Expected rhythm, stated plainly: Herdr is normally visible on this desk, so the chime should be *rare* — it exists for the narrow case where Herdr isn't up and work finishes. **The jump gesture is redefined one level up** (2026-07-16): Ctrl+Alt+J / bar-click no longer mean "jump to the longest-blocked agent" but **"take me to what most needs my attention next"** — one unified cycle, Blocked oldest-first, then Done oldest-first. A bottleneck always outranks a harvest; with anything blocked, behavior is byte-identical to #50/#52, so the redefinition is a strict extension, not a change. Done ordering uses a `doneSince` episode clock, the same pattern as `blockedSince` — MF orders the queue, Herdr still owns the states. **OS toasts are deferred, not rejected**: chime + idle badge + butler click + unified jump form v1's one coherent attention surface; a Windows notification gets added on *observed* need (completion events missed while Herdr hidden), never anticipated need. **No dwell for Done — a principled asymmetry with Blocked** (2026-07-16): Blocked's 5s dwell filters transient self-blocks, a real observed hazard; Done has no transient class — an unattended done cannot clear itself (spike: a turn ending under existing engagement skips `done` entirely), so a dwell would add latency and filter nothing. **The one-shot chime is evaluated exactly once, at the transition poll** — Herdr-foreground at that moment *consumes* it, never defers it; a later alt-tab away does not ring a bell for work you already had on screen. Routine recurring agents are not special-cased — every completion gets the same single soft cue; tuning or an opt-out waits for dogfooding evidence of noise, not anticipation of it. **"Why it's done" enrichment is cut on principle, not effort** (for now): announcing *what* completed ("Issue #N", "a batch", "a review") requires issue/batch semantics, and MF's standing boundary is that it learns no git/session/issue semantics — that knowledge lives in Control Room, which launched the pane. If enriched completions ever happen, CR pushes meaning down to a dumb MF text display (the existing "MF renders the message as text only" contract), MF never grows a brain about issues. See the fleet-awareness vision arc (PRD #44). Full rationale in ADR 0006. The consequence accepted with eyes open: reading a done pane without engaging doesn't clear its signal — "waiting to be harvested" is still true until you act.
- **Capture history — a bounded in-memory ring per verb (2026-07-19, #93–#96)**: **amends the "keeps no history" refusals below.** Captures are one-shot today — every part of a session is a local in `runSendSession` and is garbage the moment it settles — so the previous screenshot is unreachable. Each verb now keeps a ring of its last ~10 captures, arrow-navigable with Left/Right in the picker, any entry deliverable. Bounded by entry count **and** cumulative bytes (clipboard images run to tens of MB; a count-only cap would balloon memory). **Nothing new is written to disk and nothing survives a quit** — so *"No Save verb"* stands fully intact, and only *"keeps no history"* / *"no clipboard history"* move. **MF still never watches the clipboard**: passive capture was considered and rejected because it would sweep up password-manager contents and render them in the preview pane (making Windows' `ExcludeClipboardContentFromMonitorProcessing` / `CanIncludeInClipboardHistory` conventions mandatory — which Electron may not even surface, the same class of assumption that `readBuffer("CF_HDROP")` already burned us on), because Electron has no clipboard-change event on Windows so it would mean a permanent poll loop, and because Win+V already covers general clipboard history. The Relay *"never watches, polls, or logs"* clause is therefore **unamended** — only history-keeping moves. **A ring per verb, never shared** — a Relay picker must never arrow onto a screenshot. **Entries are stateful**: a crop mutates the entry in place, so arrowing is pure navigation and never destroys work; each entry keeps its pre-crop original for Esc's two-stage undo. Rejected: immutable entries (arrowing silently discards a fresh crop) and crop-pushes-a-new-entry (spends a ring slot per crop iteration). **Ring membership is the only TTL exemption** — narrowing, not discarding, "nothing is TTL-exempt": retention is bounded by the ring, and because the ring is in-memory the exemption evaporates on quit, so the next sweep reclaims everything with no persistent bookkeeping. **A fresh payload id is minted per delivery** with the path held stable — a correctness requirement, not an optimisation: the ledger keys on `(id, injectText, target)`, so re-delivering to a pane an entry already reached would return the *cached* outcome and report "Delivered, sir." while `herdr agent send` was never invoked. Arrow keys are `globalShortcut` accelerators scoped to the picker's lifetime (the overlay is `focusable: false` and handles no keys at all), entering through a fifth injected source beside `CropSource`/`AgainSource`/`RowClickSource`/`CancelSource`. Discovered while designing this and fixed first as its own slice: **`sweepExpiredCaptures` had never once been called** — 286 files / 417.6 MB of screenshots accumulated over five days in `%TEMP%\MistrFlowCaptures`, a privacy exposure rather than merely a disk leak. Full rationale in ADR 0008.
- **Both lifecycle signals chime promptly, distinguished by rhythm (2026-07-19, #91)**: amends the alert grammar above. Dogfooding exposed the Done/Blocked asymmetry as a defect: Done chimed instantly while Blocked stayed silent for **four minutes** (the persistent-block ding), so the operator got immediate notification of the harmless signal and a long silence on the urgent one. The original scoping — the ding as a *missed-bottleneck alarm*, not a *state-changed notification* — assumed the operator can see the butler's posture, but MF is frequently not on screen, so the ambient channel is dark exactly when the cue matters and the block had almost always been found by hand before the ding fired. Now: **Blocked chimes at the 5s dwell crossing** (the same poll it enters `blockedTargets`, so sound and count can never disagree; a block that self-resolves inside the dwell still never makes a noise), and **the four-minute escalation is dropped outright** — one cue per state change, never repeated, never escalating, for both signals. Rejected: chime *and* nag at 4 min (doubles the sounds to catch a case the badge and `Ctrl+Alt+J` already serve, and "never nags" governs the whole surface); no dwell at all for Blocked (Done has no transient class, Blocked demonstrably does — the §5 asymmetry survives as a *dwell* asymmetry even though the *promptness* one is gone). **Blocked is foreground-gated exactly like Done**, sharing one probe: the wiring evaluates every cheap gate for both cues first and shells out to PowerShell **at most once per poll**, never once per cue. **The cues are told apart by rhythm, not pitch** — Done one soft tone (587 Hz), Blocked two quick beeps (900 Hz) around 60 ms of silence; the gap is load-bearing because `[console]::beep` is synchronous and back-to-back calls render as one continuous tone. The old 900-vs-587 single-beep pair had already failed in use: pitch alone is not discriminable from another window, which is the only situation either cue exists for. **Blocked outranks Done when both fire on one poll** and Done stays silent that poll — a bottleneck outranks a harvest, and two racing beep shell-outs interleave into noise; the swallowed completion is the same accepted edge as foreground suppression. Config gains `blockedChime`, with the pre-#91 `persistentBlockDing` **still honoured on read** — configs are per-machine at `%APPDATA%`, outside the repo, so a silent rename would have un-silenced the cue on any machine that had already turned it off. Everything audible is asserted at the `execFile` seam against the rendered command, never by ear, which is what makes the cue testable with no sound device at all. Full rationale in ADR 0007.
